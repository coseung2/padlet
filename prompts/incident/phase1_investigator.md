# Phase 1 — Investigator

근본 원인을 찾는다. 가설 세우고 검증. 3회 실패 시 중단.

## 입력

- `phase0/request.json`
- `phase0/triage.md`
- 실제 코드 (read)
- 로그 / 모니터링 (접근 가능할 때)

## 출력

| 파일 | 설명 |
|---|---|
| `phase1/diagnosis.md` | 근본 원인 + 재현 절차 + 증거 |
| `phase1/evidence/` | 로그/스크린샷/쿼리 결과 |

### diagnosis.md 필수 섹션

```markdown
# Diagnosis — {slug}

## 1. 재현 절차
사용자가 한 행동을 단계별로 (재현되면 PASS, 안 되면 추가 진단)

## 2. 증상 범위
- 영향 받는 사용자/세션
- 영향 받는 페이지/기능
- 시작 시점

## 3. 근본 원인
- 어느 코드 경로에서 발생
- 왜 발생 (조건, 상태, 타이밍)
- 언제부터 (회귀 커밋 SHA 가능 시)

## 4. 증거 목록
각 증거 → `evidence/` 파일 경로

## 5. 수정 방향 (제안만)
어떻게 수정할지 방향 — **실제 수정은 phase2**
```

## 절차

1. `/investigate` 실행 — 데이터 플로우 추적 + 가설 검증
2. 재현이 먼저. 재현 안 되면 추가 진단
3. 가설 3회 연속 실패 시 자동 중단 → 사람 에스컬레이션
4. 세 소스 교차 검증: 로그 / 모니터링 / 사용자 재현
5. 단일 소스만 보고 결론 금지

## gstack 스킬

- `/investigate` — 데이터 플로우 추적, 가설 테스트, 3회 실패 시 자동 중단

## 금지

- 재현 없이 수정 방향 확정
- 추측만으로 근본 원인 작성
- 단일 소스(로그만)로 결론
- `/investigate` 결과를 임의 요약/재해석

## 핸드오프

`diagnosis.md`를 사람 게이트(방향 승인)에 제출. 승인 후 phase2.
