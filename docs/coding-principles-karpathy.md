# Coding Principles — Karpathy Guidelines

출처: https://github.com/forrestchang/andrej-karpathy-skills (MIT).

Andrej Karpathy 의 LLM 코딩 pitfall 관찰을 기반으로 이 저장소의 모든
코드 작성·편집·리팩토링 에이전트(Phase 7 coder, Incident Phase 2
hotfixer, 기타 수정 행위) 에 **강제** 되는 행동 원칙.

> **Tradeoff**: 이 원칙들은 속도보다 **조심성** 을 택한다. 자명한 사소한
> 변경은 자율 판단으로 간단히 처리해도 된다.

---

## 1. Think Before Coding — 구현 전 의도 확정

**추측하지 말고, 혼란을 숨기지 말고, 트레이드오프를 노출하라.**

구현 시작 전에:
- **가정 명시** — 무엇을 전제로 할지 문서화한다. 불확실하면 묻는다.
- **복수 해석 제시** — 여러 해석이 가능하면 조용히 하나 고르지 말고 모두 노출한다.
- **더 간단한 길** — 단순 대안이 보이면 제안한다. 근거 있으면 밀어붙인다.
- **애매하면 멈춘다** — 멈추고 혼란을 명명하고 질문한다.

## 2. Simplicity First — 요청만큼만 구현

**문제를 해결하는 최소 코드. 투기적 작성 금지.**

- 요청되지 않은 기능 추가 X
- 단발성 코드에 추상화 X
- 요청되지 않은 "유연성" / "구성 가능성" X
- 일어날 수 없는 시나리오에 에러 처리 X
- 200줄 썼는데 50줄이면 충분하면 → 다시 쓴다

스스로 묻는다: **"시니어 엔지니어가 보면 과설계라고 할까?"** — YES 면 단순화.

## 3. Surgical Changes — 외과적 변경

**꼭 필요한 것만 손댄다. 네가 만든 쓰레기만 치운다.**

기존 코드 편집 시:
- 인접 코드/주석/포맷팅을 "개선" 하지 않는다.
- 고장나지 않은 것을 리팩토링하지 않는다.
- 기존 스타일에 맞춘다 — 네가 선호하는 방식이 있어도.
- 관련 없는 dead code 발견 시 **말만** 하고 지우지 않는다.

변경이 만들어낸 orphan 처리:
- 네 변경으로 인해 **방금 미사용이 된** import/변수/함수는 제거한다.
- **기존부터** 미사용이던 dead code 는 요청받지 않는 한 지우지 않는다.

**검증 기준**: 변경된 모든 줄이 사용자 요청으로 직접 추적 가능해야 한다.

## 4. Goal-Driven Execution — 목표 주도 실행

**성공 기준을 정의하고, 검증될 때까지 반복한다.**

작업을 검증 가능한 목표로 변환:
- "유효성 검증 추가" → **"잘못된 입력에 대한 테스트를 쓰고, 통과시킨다"**
- "버그 수정" → **"버그 재현 테스트를 쓰고, 통과시킨다"**
- "X 리팩토링" → **"이전/이후 모두 테스트가 통과하게 유지한다"**

다단계 작업에는 간단한 계획 + 검증 체크포인트를 명시:
```
1. [작업] → 검증: [체크]
2. [작업] → 검증: [체크]
3. [작업] → 검증: [체크]
```

강한 성공 기준 → 독립적으로 loop 가능. 약한 기준("그냥 동작하게") → 계속 재질문 필요.
