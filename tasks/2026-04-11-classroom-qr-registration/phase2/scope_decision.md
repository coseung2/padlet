# Scope Decision — classroom-qr-registration

## 1. 선택한 UX 패턴

**P1(individual-qr-card) + P2(class-code-join) + P5(text-code-fallback) 결합.**

- P1 채택 사유: 9개+ 교육 플랫폼에서 검증된 표준. 이메일/비밀번호 없이 초등학생 즉시 사용 가능. (research_pack: ClassDojo, Seesaw, Discovery Education, TinyTap, Schoology 모두 채택)
- P2 병행 사유: 학급 코드로 학급 식별 → QR 토큰으로 개별 신원. 두 레이어 분리.
- P5 병행 사유: 카메라 없는 기기 대응. ClassDojo/Seesaw 모두 텍스트 코드 대안 제공.
- P3(ephemeral PIN), P4(PDF grid)는 기능에 포함되나 별도 패턴으로 분리하지 않음. P3는 2차 task(퀴즈 통합), P4는 P1 구현의 일부.

## 2. MVP 범위

### IN (이번 task)
- Classroom 모델 + CRUD (생성/조회/수정/삭제)
- 학급 코드 자동 생성 (6자리 영숫자)
- 학생 등록 (이름 입력 → UUID 토큰 + 6자리 텍스트 코드 생성)
- 학생별 QR 코드 렌더링 (클라이언트)
- QR 카드 PDF 일괄 출력 (A4 1장 최대 30명)
- QR 스캔 랜딩 → 세션 쿠키 생성 → 학급 대시보드 리다이렉트
- 텍스트 코드 수동 입력 로그인 (QR 대안)
- 학생 대시보드 (소속 학급 → 보드 목록)
- 선생님 학급 관리 화면 (학생 명단, QR 재발급, 학생 삭제)
- Board에 classroomId 연결 (학급 소속 보드)

### OUT (이번 task 제외)
- 퀴즈 보드 통합 (QuizPlayer → Student 연결) → **2차 task**
- 학생 성적/이력 누적 뷰 → 2차 task
- 학부모 연동 / 학부모 QR → 향후
- SSO/Clever 로스터링 → 향후 (솔로 프로젝트, 불필요)
- 다중 학급 소속 (학생 1명 → 학급 N개) → 향후
- QR 유효기간/만료 정책 → 향후

## 3. 수용 기준 (Acceptance Criteria)

1. 선생님이 학급을 생성하면 6자리 영숫자 학급 코드가 자동 발급된다
2. 선생님이 학생 이름을 입력하면 고유 QR 토큰과 6자리 텍스트 코드가 생성된다
3. 학급 상세 화면에서 학생별 QR 코드가 시각적으로 렌더링된다
4. "QR 출력" 버튼 클릭 시 A4 PDF가 다운로드되며, 1장에 최대 30명 카드가 격자 배치된다
5. 학생이 QR을 스캔하면 세션 쿠키가 생성되고 학급 대시보드로 리다이렉트된다
6. 학생이 텍스트 코드를 입력해도 동일하게 로그인된다
7. 학생 대시보드에서 소속 학급의 보드 목록이 조회된다
8. 선생님이 학생의 QR 토큰을 재발급하면 기존 토큰은 무효화된다
9. Board 생성/수정 시 학급을 선택할 수 있고, 선택된 학급의 학생이 해당 보드에 접근 가능하다
10. 존재하지 않는 QR 토큰으로 접근 시 에러 페이지가 표시된다

## 4. 스코프 결정 모드

**Selective Expansion** — 기존 Board/User 구조에 Classroom 레이어를 추가하는 확장이지만, 퀴즈 통합/성적 누적은 2차로 분리하여 리스크 관리.

## 5. 위험 요소

| 리스크 | 영향 | 완화 |
|---|---|---|
| QR 토큰 유출 시 타인 접근 | 학생 세션 탈취 | 토큰 재발급 기능 + 세션 기기 제한 (1 토큰 = 1 활성 세션) |
| 카메라 권한 거부 | QR 스캔 불가 | 텍스트 코드 대안 필수 제공 |
| PDF 생성 성능 | 학생 수 많을 때 | 클라이언트 사이드 렌더링 (서버 부하 없음), 30명 이상 시 페이지 분할 |
| 기존 User 모델과 학생 모델 충돌 | 스키마 복잡도 | 학생은 별도 Student 모델 (User와 분리, OAuth 불필요) |
| SQLite 동시성 | 다수 QR 동시 스캔 | 현 단계에서 문제 없음 (솔로 프로젝트, 로컬 dev) |
