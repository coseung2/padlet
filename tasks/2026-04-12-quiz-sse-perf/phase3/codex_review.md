## Independent findings
1. `[Q1: Rules of Hooks]` Severity: `NIT`. Anchor: `src/components/QuizBoard.tsx:29-49, 51-139, 141-170`; commit `65fb78a`; commit `b3e988d`. Evidence: `useMemo(sorted)` now sits at `46-49`, before the `if (!quiz)` early return at `142-165`, and the remaining parent-component hooks (`useEffect`, `useEffect`, `useCallback`) are also above that return. Reasoning: `QuizBoard` now executes the same 14 parent hooks in the same order on both render paths (`useState` x9, `useRef`, `useMemo`, `useEffect` x2, `useCallback`), so the `b3e988d` fix is correct. The conditional `LLMModal`, `Distribution`, and `QuizUrl` hooks are in separate child components and do not affect the parent hook order.
2. `[Q2: Count-gated distribution under the repo-local insert-only flow]` Severity: `NIT`. Anchor: `src/app/api/quiz/answer/route.ts:17-37`, `prisma/schema.prisma:252-265`, `src/app/api/quiz/[id]/stream/route.ts:135-164`. Evidence: the answer route rejects an existing `(questionId, playerId)` pair at `17-23` before `db.quizAnswer.create` at `29-37`, the schema enforces `@@unique([questionId, playerId])` at `264`, and the stream only recomputes distribution when `quizAnswer.count()` changes at `143-164`. Reasoning: for well-formed `A|B|C|D` inserts, I do not see a reachable count-equal/distribution-changed case in the current code; every accepted new answer increments count exactly once and moves exactly one bucket.
3. `[Q2/Q5: malformed answer edge case]` Severity: `MINOR`. Anchor: `src/app/api/quiz/answer/route.ts:8-9, 25-37`; `src/app/api/quiz/[id]/stream/route.ts:152-156`. Evidence: the API only checks that `selected` is present at `8-9` and then persists it as-is at `29-37`, while the stream only increments buckets when `a.selected in distribution` at `152-156`. Reasoning: a non-`A|B|C|D` submission can still increase `quizAnswer.count()` and `totalAnswers`, but none of the visible buckets move. That means the count-gated logic is sound for the intended UI flow, but the server does not fully enforce the premise it depends on.
4. `[Q3: cancel race between ReadableStream.cancel() and the polling loop]` Severity: `MINOR`. Anchor: `src/app/api/quiz/[id]/stream/route.ts:33-43, 45-47, 143-148, 183-190`. Evidence: `cancel()` only flips `cancelled = true` at `188-189`; `poll()` checks that flag at entry (`45-46`) and before re-scheduling (`183`), but not after awaited DB calls such as the answer count/read at `143-148`. Reasoning: no new recurring loop is scheduled after cancel, and a timeout callback that fires after cancel exits immediately at line `46`, but one already-running poll can still finish its remaining DB work before it returns. `send()` does prevent post-cancel enqueues at `33-43`, so I do not see a client-visible state leak, but the stop is not immediate.
5. `[Q4: questions-cache invalidation assumption]` Severity: `NIT`. Anchor: `src/app/api/quiz/create/route.ts:62-83`, `src/app/api/quiz/[id]/route.ts:24-70`, `src/app/api/quiz/join/route.ts:29-66`, `src/app/api/quiz/answer/route.ts:12-45`, `src/app/api/quiz/settings/route.ts:3-35`, `src/app/api/quiz/[id]/stream/route.ts:69-85`. Evidence: in the quiz API files inspected, `QuizQuestion` rows are created during quiz creation at `create/route.ts:62-83` and then only read afterward; the stream caches that question list once at `69-85`. Reasoning: I did not find a repo-local admin update/delete path for `QuizQuestion` in the current quiz API surface, so the immutability assumption behind `questionsCache` holds within this branch's codebase.
6. `[Q5: remaining live-view correctness gap]` Severity: `MINOR`. Anchor: `src/components/QuizBoard.tsx:66-98, 170-206`; `src/app/api/quiz/[id]/stream/route.ts:87-117`. Evidence: `QuizBoard` listens to `quiz-status`, `players`, `answers`, and `finished`, but not `question`; the stream emits `question` when `currentQ` changes at `96-117`, while `quiz-status` is only emitted when the status string changes at `87-94`. Reasoning: a board view that did not initiate `PATCH /api/quiz/[id]` itself can miss same-status question advances, leaving `curQ` stale while new `answers` deltas arrive for the next question.

PASS. The `b3e988d` Rules of Hooks fix is correct, the `f471946` delta/count hotfix is sound for the repo-local insert-only quiz flow, and I did not find a repo-local question-mutation path that invalidates the questions cache.
